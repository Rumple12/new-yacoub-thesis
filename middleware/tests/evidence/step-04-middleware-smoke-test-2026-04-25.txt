## Valid POST /sensor-event result

Command:

```powershell
$body = @{
  sensor_id = "temp_sensor_1"
  timestamp = "2026-04-25T20:00:00Z"
  type = "temperature"
  value = 31.4
  unit = "C"
} | ConvertTo-Json

Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/sensor-event" -ContentType "application/json" -Body $body
```

Observed result:

```text
status   message                             event
------   -------                             -----
accepted sensor event received by middleware @{sensor_id=temp_sensor_1; timestamp=2026-04-25T20:00:00Z; value=31,4; ...
```

Result summary:

- HTTP status: `202`
- Middleware accepted the temperature event.
- Event source: `temp_sensor_1`
- Event type: `temperature`
- Event value: `31.4 C`

## n8n forwarding note

n8n forwarding is intentionally skipped unless `N8N_WEBHOOK_URL` is configured.

This is expected for Step 4 because the n8n workflow is not implemented until
Step 6. The middleware bridge has a configurable placeholder, but this smoke
test did not connect to a real n8n workflow.

## Malformed POST /sensor-event result

Command:

```powershell
$badBody = @{
  sensor_id = "temp_sensor_1"
} | ConvertTo-Json

Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/sensor-event" -ContentType "application/json" -Body $badBody
```

Observed result:

```text
Invoke-RestMethod : {
  "error": "missing_required_field",
  "missing": [
    "timestamp",
    "type",
    "value",
    "unit"
  ],
  "message": "Step 4 uses a minimal inline check; JSON schemas come in Step 5."
}
```

Result summary:

- HTTP status: `400`
- Missing fields were reported.
- The response explicitly notes that Step 4 uses a minimal inline check and
  JSON schemas come in Step 5.

## Evidence conclusion

The Step 4 middleware skeleton ran locally on PC with simulated sensor and fan
state. All required initial endpoints responded, valid sensor input was
accepted, simulated fan actions worked, and malformed sensor input returned
HTTP `400`.
