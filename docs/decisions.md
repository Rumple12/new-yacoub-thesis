# Architecture decisions



This file locks the major architecture and scope decisions for the **new-yacoub** thesis plan.



These decisions are considered active unless explicitly replaced by a newer decision entry.



---



## D-001 — new-yacoub scope is the active implementation mode



**Decision:**

The repository follows the **new-yacoub** thesis plan.



**Meaning:**

- Yacoub's thesis is the full delivery target.

- Obid's original scope is reduced to the minimum integration layer required for:

- workflow validity

- shared contract definition

- validation/safety

- evaluation support

- future handoff



**Reason:**

The original broader Yacoub/Obid scope is too large for the available time and would create unnecessary implementation risk.



**Impact:**

All implementation, planning, and evaluation choices must prioritize Yacoub-complete over Obid-complete.



---



## D-002 — development starts on PC with simulated sensor data



**Decision:**

Initial development will happen on a normal PC using simulated sensor data.



**Reason:**

This reduces early implementation risk and allows the workflow, middleware, contract, and metrics pipeline to be developed in a controlled environment before involving hardware.



**Impact:**

- Early development does not depend on Raspberry Pi availability.

- Sensor data can be generated by mock scripts or simple local test endpoints.

- Real hardware integration is deferred to a later validation phase.



---



## D-003 — Raspberry Pi is a later Tier 1.5 validation step



**Decision:**

Raspberry Pi deployment is treated as a **later validation extension (Tier 1.5)**, not the initial development target.



**Reason:**

The thesis must first succeed as a complete, measurable local system before adding deployment complexity.



**Impact:**

- Local success is mandatory.

- Raspberry Pi validation is desirable but not allowed to block core thesis completion.

- The repo should remain compatible with later Pi deployment, but implementation must not begin there.



---



## D-004 — lightweight metrics collection only



**Decision:**

Metrics will be collected using lightweight local logging and Docker/runtime statistics.



**Reason:**

Heavy observability stacks such as Prometheus/Grafana introduce additional setup overhead and are outside the narrowed thesis scope.



**Impact:**

The thesis will measure:

- latency

- RAM usage

- CPU usage

- thermal data or a thermal collection path



using simple scripts, runtime statistics, and local result files.



---



## D-005 — middleware pushes events to n8n via webhook



**Decision:**

The middleware will push events to n8n using a webhook-based flow.



**Reason:**

A push-based flow is simpler for the first implementation and provides a clean event-driven architecture for the chosen scenario.



**Impact:**

- Middleware is responsible for sending sensor events into n8n.

- n8n will not poll the middleware in the first implementation.

- The first contract and evaluation flow should assume push-based communication.



---



## D-006 — one LLM/provider is an implementation parameter, not a thesis comparison axis



**Decision:**

The agent-enhanced workflow will use one cloud-hosted LLM/provider, but the exact model/provider is treated as an implementation parameter.



**Reason:**

The thesis is about workflow behavior, system integration, safety, and resource/latency characteristics — not broad model benchmarking.



**Impact:**

- The report should not drift into comparing many LLMs.

- Model/provider choice should be documented but kept narrow.

- The evaluation compares system modes, not model families.



---



## D-007 — one end-to-end scenario is the core thesis artifact



**Decision:**

The thesis will focus on one complete end-to-end IoT scenario.



**Current intended scenario:**

- temperature event

- middleware push into n8n

- workflow decision

- validation / approval path

- fan action

- metrics logging



**Reason:**

One complete measurable system is more valuable than many incomplete subsystems.



**Impact:**

Additional devices, scenarios, or workflows are deferred unless the core scenario is already complete and stable.



---



## D-008 — deterministic baseline and agent-enhanced path must both exist



**Decision:**

The repository must support both:

1. a deterministic baseline path

2. a minimal agent-enhanced path



**Reason:**

The thesis includes a comparison between a baseline and an agent-enhanced version of the same task.



**Impact:**

Implementation and evaluation must preserve both paths so the comparison remains measurable and reportable.



---



## D-009 — contract boundary is mandatory



**Decision:**

The workflow side and middleware side must communicate through an explicit shared contract.



**Reason:**

This prevents schema drift and keeps the system reproducible and reportable.



**Impact:**

Any change affecting communication must update:

- `shared_interfaces/json-schema/`

- `shared_interfaces/examples/`

- `docs/architecture/`



---



## D-010 — unsafe actions may not bypass validation



**Decision:**

No risky or malformed workflow output may directly trigger hardware-facing middleware behavior.



**Reason:**

The system must demonstrate a minimal but valid safety/approval layer.



**Impact:**

Any risky physical action must pass through:

1. validation rules

2. approval logic if needed

3. middleware execution



---



## D-011 - local n8n baseline uses Docker Compose with default local storage

**Decision:**

The first local n8n baseline uses Docker Compose with one pinned n8n container
image: `n8nio/n8n:1.123.37`.

The first baseline uses n8n's default local SQLite-backed storage persisted in
a Docker volume.

**Reason:**

This is the smallest reproducible local baseline for Step 3 and keeps the
project focused on Yacoub's measurable workflow runtime before adding
middleware, workflows, or hardware complexity.

**Impact:**

- PostgreSQL is deferred unless the first baseline proves it is necessary.
- Prometheus/Grafana are not part of the baseline.
- MCP is not part of phase 0.
- Raspberry Pi validation remains a later Tier 1.5 step.



---



## D-012 - Raspberry Pi validation is optional Tier 1.5 evidence

**Decision:**

Raspberry Pi validation remains optional Tier 1.5 evidence. The PC-first local
system is the primary thesis baseline.

**Reason:**

The narrowed thesis must not be blocked by hardware availability or deployment
complexity. Raspberry Pi validation is useful for edge-computing discussion, but
the local system already carries the core implementation and evaluation path.

**Impact:**

- Pi validation must not redesign middleware or workflows.
- Pi validation must not add multiple devices or a heavy observability stack.
- If Pi validation cannot be completed, local PC results remain valid as the
  main thesis evidence.
- No Pi results may be invented.



---



## Out-of-scope decisions currently locked



The following are explicitly out of scope unless later reopened:



- broad multi-agent systems

- elaborate ReAct expansion

- multiple memory strategies

- multiple devices/models

- deep n8n core modification

- broad guardrail research

- heavy observability stacks

- MCP in phase 0
